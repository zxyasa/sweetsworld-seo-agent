### 关键发现

## 瘦身步骤（按影响排序）

### Phase 1: 清理 draft + trash（预计省 1-2 GB）
| 步骤 | 操作 | 预估节省 |
|------|------|---------|
| 1a | 清空 trash（16 个产品）| ~50 MB |
| 1b | 审查 1,732 个 draft 产品 | |
| 1b-i | 导出 draft 产品列表（ID + 名称 + 图片数）| — |
| 1b-ii | 分类：有图 vs 无图 / 有 SKU vs 无 SKU | — |
| 1b-iii | 永久删除确认不需要的 draft（WC API batch delete）| ~1.4 GB |
| 1b-iv | 运行 WP-CLI 或插件清理孤儿 attachment（图片在 media 但没被任何产品引用）| ~0.5 GB |

### Phase 2: 清理僵尸产品（预计省 0.2-0.5 GB）
| 条件 | 数量 | 操作 |
|------|------|------|
| 已发布 + 0 销量 + 1 年没改 | ~70 | 改为 draft 或下架 |
| 已发布 + 无图片 | 21 | 补图或下架 |

### Phase 3: 缩略图清理（预计省 1-2 GB）
- WooCommerce 默认生成 6-7 种尺寸缩略图
- 某些尺寸可能没用到（取决于主题）
- 用 **Regenerate Thumbnails** 插件删掉没用的尺寸
- 需要先确认 Flatsome 实际用哪些缩略图尺寸

### Phase 4: Converter for Media WebP 管理
- Converter for Media 跑完后会在 `/uploads-webpc/` 下生成 WebP 副本
- 50,004 文件 × WebP = 额外 2-4 GB
- 如果不需要给所有图片生成 WebP（比如 draft 产品的图），可以排除
- 设置：只转换 publish 产品的图片

### Phase 5: 数据库清理（预计省 0.5-1 GB）
| 操作 | 预估节省 |
|------|---------|
| 删除 post revisions（`wp_posts WHERE post_type='revision'`）| 200-500 MB |
| 清理 transients（`wp_options WHERE option_name LIKE '_transient_%'`）| 50-100 MB |
| 删除 spam 评论 | 10-50 MB |
| 清理 WooCommerce session data | 50-200 MB |
| 优化表（`OPTIMIZE TABLE`）| 释放碎片空间 |

## 安全原则

1. **删除前先导出** — 产品 CSV + 图片 URL 列表
2. **分批操作** — 每次不超过 100 个产品
3. **先 trash 再永久删除** — 给 30 天缓冲期
4. **不删有销量的产品图片** — 即使产品下架，图片可能被博客引用
5. **备份数据库** — 大规模操作前 mysqldump

## 2026-04-11 调查结论

### 数据库分析结果
| 分类 | 数量 | 能删？ |
|------|------|--------|
| Unattached 总数 | 4,443 | ⚠️ |
| 其中被产品引用的 | 3,909 (88%) | ❌ 不能删！|
| 真正无产品引用的 | 534 (12%) | ⚠️ 可能被博客/页面引用，无法确认 |
| Parent = published | 1,640 | ❌ |
| Parent = draft | 769 | 跟 draft 产品一起处理 |
| Parent = trash | 12 | ✅ 可删 |

### 关键发现
- **88% 的 unattached 图片其实被产品在用** — unattached ≠ 孤儿，不能按此条件删
- 534 个"真孤儿"因数据库 LIKE 查询超时无法验证是否被博客/页面引用
- 人工过 4,500 张不现实

### 决定：暂不清理
- 磁盘 148GB 用了 93GB，还有 **49GB 空余（33%）**
- PSI 已达 92/98 分，Converter for Media 全站 WebP 生效
- **盲删风险 > 收益**
- 等磁盘用到 80%+ 再回来处理

### 未来触发条件
当以下任一条件满足时重新执行清理：
1. 磁盘使用 > 120GB（80%）
2. 大规模下架产品（一次性下架 100+ 产品）
3. 需要迁移服务器/降低备份成本

### 到时候的安全做法
1. 先清 trash 产品（最安全）
2. 审查 draft 产品（人工判断哪些永远不发布）
3. 用 WP 插件（如 Media Cleaner Pro）扫描孤儿图 — 它会检查所有 post_content、meta、widget、shortcode
4. 不要自己写脚本删——用专业插件，它知道检查哪些地方
